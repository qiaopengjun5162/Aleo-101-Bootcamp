## 问题

**Q1. Aleo 被定义为首个提供哪三个特性的去中心化应用平台？**

A:  隐私性（Private）、可编程性（Programmable）、可扩展性（Scalable）

---

**Q2. 在去中心化账本中，为什么"重复执行"既是核心优势也是主要弱点？**

A:
- **作为优势：** 所有节点对同一笔交易重复执行并独立验证结果，这正是去中心化信任的根基——无需信任任何单一节点，只要多数节点诚实，账本就不可篡改、可公开审计，从而实现 trustless 的共识。
- **作为弱点：**
  1. **扩展性差。** 每个节点都要重新执行全网所有交易，全网吞吐被单节点性能上限锁死，无法水平扩展。
  2. **隐私丧失。** 为了让任意节点都能独立验证，交易输入、状态、合约逻辑必须以明文公开，用户的资产、行为、身份因此完全暴露在链上。

Aleo 的解决思路就是用 ZK 证明替代"重复执行"：用户在本地执行并生成简短证明，节点只验证证明本身，从根本上解耦了"验证"与"暴露数据"。

---

**Q3. Aleo 采用的记录模型（Record Model）与传统的账户模型相比，最大的优势是什么？**

A: 最大的优势是 **默认隐私 + 链下执行可扩展**：

- **账户模型（如以太坊）**：所有账户余额、合约状态都以明文存在全局状态树中，任何人都能查询任何地址的资产与行为，隐私完全暴露；同时每个节点必须重新执行每一笔交易才能更新状态。
- **Aleo 的记录模型（UTXO-like + 加密）**：状态以加密的 record 形式存在，链上只能看到 record 的承诺（commitment）和花费时的 nullifier，**发送方、接收方、金额、合约调用参数都是密文**，只有持有 viewing key 的人才能解密。同时交易在用户本地执行并生成 ZK 证明，节点只验证证明，不需要重放计算——隐私和可扩展性一并解决。

---

**Q4. Aleo 开发 Leo 语言的初衷（设计目标）是什么？**

A: Leo 的设计目标是 **让普通开发者不需要懂底层电路和密码学，就能写出正确、安全的 ZK 应用**：

1. **抽象掉电路细节。** 传统 ZK 开发需要直接面向 R1CS / 约束系统手写电路，门槛极高；Leo 提供 Rust-like 的高级语法，编译器自动把高级代码编译为底层约束。
2. **静态类型 + 形式化语义。** 借鉴 Rust 的类型系统，让代码可读、可审计、可形式化验证，降低写出"看似正确但电路有漏洞"的风险。
3. **专为 ZK 应用而生。** 内建 record、transition、mapping 等 Aleo 语义原语，天然适配链下执行 + 链上验证的模型，而不是把 ZK 当作通用语言的附加库。
4. **工程化工具链。** 提供 `leo new / build / run / test` 等命令、包管理、测试框架，让 ZK 开发体验接近现代主流开发语言。

一句话：**把 ZK 从"密码学家的工具"变成"普通开发者的能力"。**

---

**Q5. 相比其他隐私方案，Aleo 在证明上有哪些显著优势？**

A:
1. **通用可编程，而非单一场景。** Zcash 等早期方案的 ZK 只用于"屏蔽转账"这一固定逻辑，Aleo 则把 zk-SNARK 用在通用图灵完备的合约执行上，可以承载任意 dApp，而非仅仅是混币。
2. **链下执行 + 链上验证。** 用户在本地生成证明，节点只做轻量验证，避免了"重复执行"瓶颈；这也意味着合约执行复杂度对链上成本几乎无影响——和 zk-rollup 的思路类似，但 Aleo 把它内建到 L1。
3. **证明小、验证快。** 采用 Marlin / Varuna 等 SNARK 方案，证明大小为 KB 级、验证时间为毫秒级，适合链上验证。
4. **默认隐私（Privacy by Default）。** 隐私不是可选项而是基础设施层默认行为，避免了"屏蔽池过小 → 反匿名化"的尴尬，从设计源头解决匿名集问题。
5. **合规友好。** 通过 viewing key 等机制可对授权方有选择地披露数据，兼顾隐私与合规审计需求，这是纯混币方案难以提供的能力。
