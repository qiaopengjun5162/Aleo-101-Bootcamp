# DevDocs 贡献

欢迎来到 Aleo DevDocs 贡献指南。我们邀请你一起改进开发者文档，让它变得更完整、更新更及时。无论贡献大小，都会帮助整个 Aleo 开发生态变得更好。

本页会说明如何有效地参与文档贡献。

## 提供反馈

如果你发现了问题，或者有改进建议，但暂时没有时间亲自修改，也可以通过 GitHub Issues 提供反馈。

建议流程：

1. 打开 `welcome` 仓库的 Issues 页面。
2. 点击 `New Issue`。
3. 如果仓库提供合适的 issue 模板，就优先使用；否则也可以从空白 issue 开始。
4. 使用清晰简洁的标题概括问题或建议。
5. 在描述中补充必要信息：
   对于 bug 或错误：说明你原本预期看到什么、实际看到了什么，并在适用时写出复现步骤。
   对于改进建议：说明你的想法，以及它为什么能提升文档质量。
   如果有帮助，也可以附上截图、示例代码或链接。
6. 添加合适的标签，例如 `documentation`、`enhancement`、`bug`。
7. 提交 issue。

这些反馈可以帮助维护者更快定位问题，并让文档保持准确、可用。

## 开始贡献

1. Fork `welcome` 仓库。
2. 在本地克隆你自己的 fork：

```bash
git clone https://github.com/YOUR-USERNAME/welcome.git
cd welcome
```

3. 进入文档目录：

```bash
cd documentation
```

这里就是主要的文档内容所在位置。

## 修改文档

1. 为你的修改创建一个新分支：

```bash
git checkout -b your-feature-branch
```

2. 修改对应文档文件。大部分内容都采用 Markdown 格式。
3. 使用清晰的提交信息提交改动：

```bash
git commit -am "Add description of your changes"
```

## 提交贡献

1. 把改动推送到你的 fork：

```bash
git push origin your-feature-branch
```

2. 回到 `welcome` 仓库，从 `your-feature-branch` 创建新的 Pull Request。
3. 用清晰的标题和描述说明这次修改的内容。
4. 提交 PR，等待审查。

## 建议

- 保持表达清晰、简洁，并尽量延续现有文档风格。
- 正确使用 Markdown 的标题、代码块和其他格式。
- 如果新增页面，记得同步更新侧边栏配置。
- 如果某些内容更适合配图或流程图，可以补充必要的图片。
- 提交前检查所有链接是否正确可用。

## 需要帮助？

如果你在贡献过程中遇到问题，可以在 `welcome` 仓库中继续提 issue，或联系维护者。

感谢你为 Aleo 文档做出贡献。
